# CopyRoom Concept Review

**Reviewer:** Claude (concept review agent)
**Date:** 2026-05-28
**Documents reviewed:**
- `COPIER_COPYROOM_CONCEPT.md` — Original workshop-only concept
- `COPYROOM_CONCEPT.md` — Expanded concept with installable CLI/toolkit

**Review scope:** Architecture coherence, scope risks, conceptual gaps, build-order feasibility, and practical friction points.

---

## Overall Assessment

The CopyRoom concept is **strong, well-structured, and unusually self-aware about its own risks**. The separation of concerns (workshop vs. CLI vs. template repos vs. shared tooling vs. generated projects) is the right architecture. The concept documents are thorough enough to build from, and the anti-patterns list demonstrates real operational experience with template management pain.

The evolution from Doc 1 (workshop-only) to Doc 2 (workshop + installable CLI) is a natural and correct progression. The project-side problem is real: generated projects today have no structured way to understand their own template lineage, and agents working inside generated projects have no reliable affordances for safe template updates.

That said, there are several areas where the concept needs stress-testing, tightening, or deliberate scoping decisions before implementation begins.

---

## Strengths

### 1. Clean responsibility boundaries

The four-entity model (workshop, CLI, template repos, generated projects) with shared tooling as a fifth entity is exactly right. The concept is disciplined about not collapsing these into each other. The rule "CopyRoom coordinates template work. Template repos contain template source. Generated projects contain local project choices." is a durable principle.

### 2. Workshop-as-coordination, not workshop-as-monorepo

Rejecting the monorepo model and keeping template repos independent is the correct call. The concept explains why clearly and offers practical alternatives (local clones, optional lock file) without over-engineering the repo management layer.

### 3. Copier answers as authoritative state

The decision to treat `.copier-answers.yml` as the source of truth for Copier state and `copyroom.project.yml` as advisory workflow metadata is critical. Many tools fail by trying to own state they should only read. This concept gets the ownership boundary right.

### 4. Selective golden testing

The golden output strategy is pragmatic. Snapshotting directory trees and key config files rather than every generated byte avoids the brittleness trap that kills snapshot-heavy test suites.

### 5. Agent-first design without agent-only design

The concept treats agents as first-class consumers of the CLI's structured output without making CopyRoom useful only to agents. The `--json` flag pattern, context roots, agent briefs, and tool manifests are concrete affordances. The human-readable defaults keep the tool usable for maintainers who aren't running agents.

### 6. Explicit anti-patterns

The anti-patterns sections in both documents are genuinely useful guardrails. They show awareness of the failure modes that template management systems typically fall into (over-prompting, generated file bloat, hidden runtime dependencies, branch-per-project strategies).

## Concerns and Risks

### 1. Scope creep is the primary existential risk

**Severity: High**

Doc 2 describes a system with 5 CLI modes, ~50 commands across 8 command groups, Pydantic models for config, structured JSON output, devenv integration, context discovery with search, agent context packs, release automation, and a 5-phase MVP.

This is a lot of surface area for a v0.1.0. The concept acknowledges "start small" but then defines a target that is not small. The risk is that building toward the full vision delays the point where CopyRoom becomes useful.

**Recommendation:** Define a brutally minimal "Phase 0" that delivers value within a week. This should be:
- Mode detection (workshop vs. project vs. standalone)
- `copyroom project inspect` (read `.copier-answers.yml` + optional `copyroom.project.yml`, print summary)
- `copyroom template status` (compare recorded template ref to latest available tag)

Everything else should be gated behind demonstrated need, not speculative architecture.

### 2. The CLI/workshop coupling is underspecified

**Severity: Medium**

Doc 2 presents two options for where the CLI lives (inside CopyRoom repo vs. separate `copyroom-tool` repo) and recommends starting with Option A. But the concept doesn't address what happens when workshop-mode commands need workshop-specific files (registry, scenarios, golden outputs) while the CLI is installed as a standalone tool in a generated project.

If the CLI is a single `uv`-installable package, it needs to carry workshop-mode logic even when installed in a project that has no workshop. This means the installed package will contain dead code paths in project mode, and workshop mode won't work unless you're inside a CopyRoom checkout.

**Recommendation:** Make the separation explicit in the package design. Either:
- Ship one package with mode-gated commands (acceptable for now, but document it clearly)
- Ship a core package (`copyroom`) and an optional workshop extension (`copyroom[workshop]`) — more complex but cleaner

The concept should state which approach is intended and why.

### 3. Config schema versioning needs more thought

**Severity: Medium**

The concept uses `copyroom: version: 1` in project manifests but doesn't describe what happens at version 2. Will v2 configs be backward-compatible? Will the CLI need to read both? Will templates that generate `copyroom.project.yml` need to know which config version the user's installed CLI supports?

This is a real problem because the config file is generated by templates but consumed by a separately-versioned CLI. Template authors and CLI maintainers can easily get out of sync.

**Recommendation:** Add a brief section on config evolution strategy. At minimum:
- State that v1 fields will not be removed in v2 (additive-only changes)
- Or state that the CLI will support reading older config versions
- Define who is responsible for config migration (CLI? template? user?)

### 4. Context discovery is a rabbit hole

**Severity: Medium**

The context discovery system (`copyroom context roots`, `copyroom context search`, `copyroom context agent-pack`) is appealing but potentially unbounded. The concept defines default context roots per project type, which is useful. But `copyroom context search` is essentially "build a project-aware search engine," and `copyroom context agent-pack` is "generate a multi-file briefing."

These are valuable features, but they're also the kind of features that consume enormous implementation effort relative to their core contribution. An agent that can read files and run grep doesn't strictly need a custom search wrapper.

**Recommendation:** Defer context search and agent-pack to Phase 3+ as planned, but consider whether `copyroom context roots` (just listing the configured paths) plus `copyroom agent brief` (a static summary, not a generated pack) is sufficient for the first few months. Agents like Claude Code already have file reading, glob, and grep capabilities — they mostly need to know *where to look*, not a custom search layer.

### 5. Devenv integration may be premature

**Severity: Low-Medium**

The devenv commands (`copyroom devenv status`, `copyroom devenv shell-info`, `copyroom devenv locate-bin`, `copyroom devenv dependency-source`) add a significant surface area for a tool that is fundamentally about Copier template management. Devenv is an orthogonal concern — it matters for some generated projects but has nothing to do with template rendering, updating, or releasing.

**Recommendation:** Keep devenv as a config field in `copyroom.project.yml` (useful for context), but defer all `copyroom devenv *` commands. Projects that use devenv already have `devenv` CLI. CopyRoom should focus on the template lifecycle and not become a general-purpose project introspection tool.

### 6. The scenario model doesn't account for multi-template composition

**Severity: Low-Medium**

The concept handles single-template scenarios well but doesn't address what happens when a generated project uses multiple templates or partial templates layered on top of each other. For example:

```text
1. Generate base project from python-cli-template
2. Layer pytest-harness-template on top
3. Layer github-actions-ci-template on top
```

Copier supports this via multiple `.copier-answers.yml` entries or separate answer files, but the CopyRoom scenario model assumes one template per render. The project manifest also assumes a single `template_id`.

**Recommendation:** Acknowledge this limitation explicitly. Either:
- State that CopyRoom v1 targets single-template projects only
- Or sketch how multi-template composition would work (multiple template entries in the project manifest, ordered update sequences, per-template scenarios)

### 7. The "shared tooling" concept needs sharper edges

**Severity: Low**

The concept distinguishes between templates and shared tooling, which is correct. But the boundary is defined mostly by examples rather than by a crisp rule. When does something stop being a template and become shared tooling? The concept says "use shared tooling when logic is reusable across many generated projects" — but a template is also reusable across many generated projects.

The real distinction seems to be:
- **Template:** generates files at copy-time, then Copier's job is done
- **Shared tooling:** referenced at runtime by generated projects (imported, included, fetched)

**Recommendation:** Sharpen the definition. Something like: "A template produces files. Shared tooling is consumed by files." This makes the boundary mechanical rather than judgmental.

### 8. Release automation spans too many repositories

**Severity: Low**

The release workflow touches the template repo (tag), the CopyRoom workshop (lock file, golden updates), and potentially downstream generated projects (update notifications). Coordinating releases across multiple independent repos is inherently messy.

The concept describes the steps clearly, but `copyroom release tag` will need to push tags to a remote template repo while also committing changes to the CopyRoom workshop repo. This cross-repo coordination is where things tend to break in practice.

**Recommendation:** For v1, keep release as an advisory check (`copyroom release check`) and let humans do the actual tagging. Automated cross-repo releases can come later once the check-and-validate layer is trusted.

## Gaps and Missing Pieces

### 1. Error handling philosophy

Neither document describes what happens when things go wrong. What does the CLI do when:
- `copier update` fails mid-way?
- A post-update check fails?
- The template source repo is unreachable?
- The project manifest is malformed?
- The git worktree is dirty when clean is required?

The concept should state an error philosophy: does the CLI try to recover, roll back, or just report and exit? For a tool that wraps destructive operations (template updates can overwrite files), this matters.

**Recommendation:** Add a section on error handling. The safest default: report clearly and exit. Do not attempt automatic rollback — that's more dangerous than a clear failure message.

### 2. Testing strategy for the CLI itself

The concept describes testing templates extensively (render tests, update tests, golden tests, matrix tests) but says little about testing the `copyroom` CLI itself. The recommended initial file set includes `test_models.py`, `test_discovery.py`, and `test_project_inspect.py`, but there's no strategy for how to test the CLI's interaction with real Copier templates, git repos, and file systems.

**Recommendation:** Add a brief section on CLI testing. Key question: will CLI tests use real Copier templates (slow, realistic) or mock everything (fast, fragile)? A practical approach: unit-test models and config loading, integration-test key commands against a minimal fixture template.

### 3. Upgrade path for the CLI itself

The concept describes how generated projects upgrade their templates, but not how they upgrade the `copyroom` CLI. If `copyroom` is a dev dependency in generated projects, what happens when a new CLI version changes the config schema, adds new commands, or deprecates old ones?

**Recommendation:** Add a note on CLI versioning and upgrade expectations. Since `copyroom.project.yml` is advisory metadata, CLI upgrades should be non-breaking for existing project manifests. New config fields should have defaults. Old commands should not be removed without deprecation.

### 4. Permissions and security model

The CLI wraps `copier update`, creates git branches, runs arbitrary commands from config files, and may push tags. There's no discussion of what trust boundaries exist. A malicious `copyroom.project.yml` could define `post_template_update` commands that run harmful code.

**Recommendation:** Add a brief security note. The practical stance: CopyRoom trusts local config files the same way `Makefile`, `justfile`, and `package.json` scripts are trusted. But the CLI should never fetch and execute remote code without user confirmation.

### 5. What happens to projects that stop using CopyRoom?

The concept says CopyRoom should not be a hidden runtime dependency, and that templates should remain directly usable with Copier. But if a project has been using `copyroom template update` and decides to stop, what's the off-ramp?

**Recommendation:** Add a brief note: removing CopyRoom means deleting `copyroom.project.yml` and using `copier update` directly. The `.copier-answers.yml` file is unaffected. This should be a non-event, and it's worth stating explicitly to reinforce the "not a hidden dependency" commitment.

## Document-Specific Notes

### Doc 1 (COPIER_COPYROOM_CONCEPT.md) — Workshop-Only Concept

This document is well-structured and appropriately scoped for its purpose. It reads as a solid foundation document. Minor notes:

- The `justfile` examples are good. They establish a predictable command interface early.
- The "Why Not Git Submodules" section is a good preemptive FAQ.
- The scenario naming conventions (minimal, standard, full, ci-enabled, existing-repo-update, migration, edge-case-names) are practical.
- The fixture model for update testing is well thought out.

No major issues with Doc 1 in isolation.

### Doc 2 (COPYROOM_CONCEPT.md) — Expanded CLI Concept

This document is more ambitious and consequently has more surface area for friction. Specific notes:

- The ASCII architecture diagram at the top is helpful. Keep it.
- The five CLI modes (workshop, project, template-repo, shared-tooling, standalone) are logical but may create confusion about which commands are available where. The `copyroom explain-mode` command is a smart mitigation.
- The `copyroom doctor` command is mentioned but never defined. It should either be described or removed.
- The Python code examples for Pydantic models are useful for establishing the implementation tone. They demonstrate the intended level of type safety and structure.
- The two `copyroom.project.yml` examples (Python CLI and Neovim plugin) are valuable — they show the config model is flexible enough for different project types.

## Build Order Assessment

The 5-phase MVP plan in Doc 2 is reasonable in sequence but aggressive in total scope. Observations:

**Phase 1 is correctly scoped.** Mode detection, config loading, and `project inspect` establish the foundation without requiring Copier integration.

**Phase 2 is the critical value phase.** Safe template updates are the single most useful thing CopyRoom can do for generated projects. This should be prioritized ruthlessly.

**Phase 3 (context/agent) could be deferred significantly.** Agents can work with basic project inspection and manual file reading. Context packs and search are nice-to-have.

**Phase 4 (workshop ops) is actually Phase 1 of Doc 1.** If the workshop already exists with shell scripts, the CLI versions of these commands can wait until the project-side commands prove the CLI model.

**Phase 5 (release automation) should be the last thing built.** Release is high-stakes and low-frequency. Advisory checks are enough for a long time.

**Suggested revised priority:**

```text
1. Package skeleton + mode detection + models + config loading
2. project inspect + template status (read-only, safe, immediately useful)
3. template update wrapper (the core value proposition)
4. git workflow helpers (branch creation, clean worktree checks)
5. agent brief + agent tools (structured output for LLM agents)
6. Workshop render + test commands (port shell scripts to CLI)
7. Golden diffs + update simulation
8. Context roots + context search
9. Release checks
10. Everything else
```

## Naming and Terminology

The naming is generally consistent and clear. A few observations:

- **"Workshop"** is a good term. It's distinctive and immediately communicates "this is a workspace, not a runtime system."
- **"Scenario"** is clear and well-used.
- **"Golden output"** is standard testing terminology. Good.
- **"Context roots"** is slightly unfamiliar but becomes clear in context. Acceptable.
- **"Agent brief"** is evocative and clear.
- **"copyroom.project.yml"** — the `copyroom.` prefix is appropriate for namespacing. Consider whether this file should live at the project root or in a `.copyroom/` directory. Root is simpler; a directory is cleaner if more CopyRoom files accumulate.

---

## Final Verdict

**The concept is sound and ready for implementation with scoping discipline.**

The architecture is right. The separation of concerns is right. The decision to keep template repos independent is right. The Copier-answers-as-authority model is right. The agent affordance design is right.

The primary risk is building too much too early. The concept describes a complete system, but the value curve is steep at the front (mode detection, project inspection, safe template updates) and flat at the back (context search, release automation, devenv integration). Implementation should follow the value curve, not the document order.

**Three things to decide before writing code:**

1. **Where does the CLI package live?** Option A (inside CopyRoom repo) or Option B (separate repo)? Pick one now and commit.

2. **What is the v0.1.0 command set?** Define the exact commands that constitute the first usable release. Recommended: `version`, `status`, `explain-mode`, `project inspect`, `template status`.

3. **What is the first real template to integrate?** Pick one template (likely `nvim-plugin-template` or `python-cli-template`) and make CopyRoom work end-to-end for that one template before adding others.

Everything else can be decided incrementally as the tool develops.
